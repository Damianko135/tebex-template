# Tebex integration

## Which Tebex API this app uses, and why

Tebex publishes several distinct, non-interchangeable integration surfaces. This app uses exactly
one of them for all product/catalog/basket data:

| API | What it's for | Used here? |
| --- | --- | --- |
| **Headless API** | Build a fully custom storefront UI on top of a catalog you manage in Tebex's own Webstore Builder (categories, packages, CMS pages, sidebar modules, tiers). Public account token auth. | **Yes — this is the entire `lib/tebex/` layer.** |
| **Checkout API** | Use Tebex purely as a Merchant-of-Record payment processor, with *your own* product catalog (Tebex has no concept of your categories/packages). Project ID + Private Key auth, requires compliance approval. | No. |
| `@tebexio/tebex-sdk-nodejs` (official Node SDK) | An alternative server-side client for the above APIs. | No — not a well-maintained project as of this writing, which is exactly why this app generates its own client (below) instead. |
| `@tebexio/tebex.js` | Client-side checkout-embed widget. Not a data API at all. | Yes, but only for the checkout popup — see [checkout.md](./checkout.md). |

**Why Headless, not Checkout:** this app's entire admin dashboard (`/admin/categories`,
`/admin/packages`, `/admin/pages`, `/admin/sidebar`, `/admin/tiers`, `/admin/webstore`) exists to
*browse* a catalog that lives in Tebex — there's deliberately no create/edit/delete for any of it,
because that's done in Tebex's own control panel. Switching to the Checkout API would mean this
app would need to own and store its entire product catalog itself before Tebex enters the picture
at all — a fundamentally different product, not a swap-in.

## The generated client: `tebex-headless`

Rather than hand-writing request/response types for ~30 Headless API operations, this repo
generates a fully-typed client from Tebex's own OpenAPI spec, published as a local pnpm workspace
package (`tebex/`, package name `tebex-headless`):

```text
tebex/
  openapi.yaml            fetched from tebexio/TebexHeadless-OpenAPI on GitHub
  src/api/schema.d.ts      generated by openapi-typescript — DO NOT hand-edit
  src/api/endpoints.ts      generated by tebex/scripts/build-endpoint-manifest.ts — DO NOT hand-edit
  src/client.ts             hand-written
  src/index.ts              hand-written
```

Regenerate all of it after a Tebex API change:

```bash
cd tebex
pnpm run update   # fetch-schema && generate && build-manifest
```

### Two clients, because the API has two base URLs

`tebex/src/client.ts` exports two factory functions:

- `createTebexClient(publicToken)` → base URL `https://headless.tebex.io/api/accounts/{token}`.
  Used for everything except basket-package mutations.
- `createTebexBasketClient()` → base URL `https://headless.tebex.io/api/baskets` (no account
  token in the path). The `Baskets`-tagged operations (add/remove/update basket packages) are
  declared in the spec with their own `servers` override that openapi-fetch's single-base-URL
  client can't represent in one client instance, so a second one is needed.

Both are lazily-constructed singletons in `lib/tebex/client.ts` (`getTebexClient()`,
`getTebexBasketClient()`). `getTebexClient()` returns `null` when `TEBEX_STORE_TOKEN` is unset, so
every caller has to handle the "not configured" case explicitly — see below.

### `lib/tebex/`: the only module allowed to import `tebex-headless`

- **`lib/tebex/client.ts`** — the singletons described above.
- **`lib/tebex/result.ts`** — `TebexResult<T> = { ok: true; data: T } | { ok: false; error }`, plus
  `unwrapTebexResponse()` which normalizes an `openapi-fetch` response into that shape, and
  `OperationResponse<"operationId">` — extracts a response type from the generated `operations`
  map rather than `components["schemas"]`, because openapi-typescript sometimes emits
  structurally-distinct-but-equivalent shapes for the same discriminated union depending on which
  path resolves a `$ref`.
- **`lib/tebex/errors.ts`** — `TebexApiError` (normalizes RFC 7807 problem+json error bodies) and
  `TebexNotConfiguredError` (thrown/returned when `TEBEX_STORE_TOKEN` is unset).
- **`lib/tebex/queries.ts`** / **`lib/tebex/mutations.ts`** — one thin, typed function per Headless
  API operation, each returning `Promise<TebexResult<...>>`. Every page and Server Action calls
  these, never `tebex-headless` directly.

Every function follows the same shape:

```ts
export async function getPackage(
  packageId: string
): Promise<TebexResult<OperationResponse<"getPackage">>> {
  const client = getTebexClient();
  if (!client) return notConfigured();
  return unwrap(client.GET("/packages/{packageId}", { params: { path: { packageId } } }));
}
```

Callers always check `result.ok` and render an `ErrorPanel` (which special-cases
`TebexNotConfiguredError` into a "Store not connected, set `TEBEX_STORE_TOKEN`" message) on
failure — there's no thrown-exception error path for API calls, by design.

## Basic-auth-gated operations

Two operations (`getUserTieredCategories`, `updateTier`) are declared in the spec with
`security: basicAuth` but the spec doesn't document what credentials they expect. This app doesn't
have a stored answer for that — the admin operator supplies raw username/password via form fields
(`app/admin/(protected)/tiers/tier-tools.tsx`) each time, passed straight through as an
`Authorization: Basic` header (`lib/tebex/result.ts`'s `basicAuthHeader()`).

## Extending the client

If Tebex adds a new endpoint you need:

1. Run `pnpm run update` in `tebex/` to pull the latest spec and regenerate types.
2. Add a thin wrapper function in `lib/tebex/queries.ts` or `mutations.ts` following the pattern
   above.
3. If it's a new resource type worth its own admin page, see
   [admin-dashboard.md](./admin-dashboard.md).

`app/admin/(protected)/api-explorer` (`components/admin/api-explorer.tsx`) reads
`ENDPOINT_MANIFEST` directly and needs no changes — it stays in sync automatically whenever you
regenerate.
