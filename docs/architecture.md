# Architecture

## The two applications sharing one Next.js app

`app/layout.tsx` is the only layout shared by both halves of this project:

- **`app/(store)/`** — a URL-transparent route group (`(store)` never appears in the URL) holding
  the customer-facing storefront: home, categories, packages, search, basket, checkout
  redirect/return pages, and account sign-in. Shares `components/store/storefront-shell.tsx`
  (site header/footer).
- **`app/admin/`** — the internal admin console. `app/admin/login/` is public; everything under
  `app/admin/(protected)/` is gated by Better Auth (see [authentication.md](./authentication.md))
  and shares `components/admin/admin-shell.tsx` (sidebar shell).

Both route groups sit under one root layout because they share the same theme system, fonts, and
global CSS — see [theming.md](./theming.md) for why the theme has to be injected at the very root.

## Data flow

Neither route group owns any product data. Every category, package, CMS page, sidebar module, and
tier comes from Tebex's Headless API at request time — see
[tebex-integration.md](./tebex-integration.md). This app's only two pieces of first-party,
self-owned state are:

1. **The storefront theme** (color tokens) — stored in Redis/KV, see [theming.md](./theming.md).
2. **Admin accounts and sessions** — stored in Redis via a custom Better Auth adapter, see
   [authentication.md](./authentication.md).

Everything else (baskets, coupons, gift cards, tiers) is read and mutated directly against Tebex's
API on every request; nothing is cached or duplicated locally.

## `lib/` layering

```text
lib/
  tebex/          server-only Headless API client, queries, mutations, typed errors
  store/           storefront business logic: basket cookie identity, cart Server Actions
  ui/               the theme system: tokens, Redis persistence, CSS serialization
  storage/          generic KV wrapper (unstorage) used by the theme system
  auth.ts, auth-*.ts, redis.ts   Better Auth config + its Redis-backed adapter
  action-state.ts, form-data.ts  shared Server Action plumbing
  format.ts, site.ts, utils.ts   small stateless helpers
```

The rule of thumb: **`lib/tebex/` is the only module allowed to know about Tebex's HTTP API**.
Every page and component goes through `lib/tebex/queries.ts` / `lib/tebex/mutations.ts` (or the
storefront-specific wrappers in `lib/store/`), never `tebex-headless` directly. This keeps the
generated-client-vs-hand-written-logic boundary in exactly one place.

`lib/tebex/`, `lib/store/basket-cookie.ts`, `lib/store/basket.ts`, `lib/auth*.ts`, and `lib/redis.ts`
all start with `import "server-only"` — that's a hard compile-time guarantee (via the `server-only`
package) that a Client Component can never accidentally import them and leak a token or cookie
value into client JS.

## `components/` layering

- **`components/ui/`** — shadcn + [base-ui](https://base-ui.com) primitives. Left as close to
  generated/vendored as possible; only touched when there's a concrete bug or a documented,
  narrow extension point (e.g. `Button`'s `nativeButton` auto-detection, `Progress`'s
  `indicatorClassName`/`indicatorStyle` passthrough).
- **`components/admin/`** — admin-only composition: `DataTable`, `PropertyGrid`, `JSONViewer`,
  `AdminShell`/`AppSidebar`, per-resource table components (`CategoryTable`, `PackageTable`, ...).
- **`components/store/`** — storefront-only composition: `StorefrontShell`, `PackageCard`,
  `AddToBasketForm`, `CheckoutLink` (see [checkout.md](./checkout.md)), etc.
- **`components/*.tsx`** at the root (`action-feedback.tsx`, `empty-state.tsx`, `error-panel.tsx`)
  — shared by both admin and storefront.

## The `tebex/` workspace package

`tebex/` is a second pnpm workspace member (see `pnpm-workspace.yaml`), published locally as the
`tebex-headless` package that `lib/tebex/client.ts` imports. It is **generated**, not
hand-written:

```text
tebex/
  openapi.yaml            fetched from Tebex's published OpenAPI spec
  src/api/schema.d.ts     generated by openapi-typescript
  src/api/endpoints.ts    generated by tebex/scripts/build-endpoint-manifest.ts
                          (drives the admin /admin/api-explorer page)
  src/client.ts           hand-written: two openapi-fetch clients (see tebex-integration.md)
  src/index.ts            hand-written: public exports
```

Run `pnpm run update` inside `tebex/` any time Tebex's API changes; see
[development.md](./development.md).

## Rendering model

Every route in both `app/(store)/` and `app/admin/` is `export const dynamic = "force-dynamic"`.
This is deliberate, not an oversight: every page's data (catalog, baskets, admin resources) is
live and request-specific, and there's no meaningful static shell to prerender. `loading.tsx`
exists at the top of both route groups (`app/(store)/loading.tsx`,
`app/admin/(protected)/loading.tsx`) to give instant feedback via React Suspense while each page's
server-side data fetch is in flight.

Client Components (`"use client"`) are used only where real interactivity is needed — forms with
`useActionState`, the theme color picker, the sidebar's collapse state, the embedded checkout
trigger. Everything else is a Server Component by default.
